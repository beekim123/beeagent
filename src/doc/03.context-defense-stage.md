# Context Defense 阶段记录

这个阶段主要不是继续做摘要压缩，而是在进入真正的长上下文问题之前，先补一层更轻量、更即时的“上下文防线”。

它的目标是：当对话里出现很多工具结果时，不急着做总结，而是先通过截断、预算清理和 TTL 修剪，把最容易膨胀的上下文压住。

## 本阶段做了什么

1. 新增 `src/context/defense.ts`：
   - `TokenTracker`：记录精确 token 和增量字符数，用于估算上下文压力。
   - `estimateMessageTokens()`：按消息内容粗略估算 token。
   - `truncateToolResults()`：对超长工具结果做截断和预算清理。
   - `ttlPrune()`：按消息年龄对旧工具结果做软修剪和硬清除。
   - `applyDefense()`：把上述逻辑串起来，形成统一入口。

2. `index.ts` 接入 defense：
   - 启动时不再走 MCP / 延迟工具演示路径。
   - 只保留本地工具注册。
   - 在每轮 `agentLoop` 结束后执行 `applyDefense()`。
   - defense 修改了消息数组后，会通过 `store.replaceAll()` 回写 session。

3. 新增一组 CLI 快捷命令，专门用来测试 defense：
   - `sim` / `模拟长对话`
   - `defend` / `执行防线`
   - `status` / `查看状态`

## 为什么还需要 defense

`compressor.ts` 里的摘要压缩适合处理“已经很长、必须总结”的历史。

但在实际对话里，更常见的问题是：

1. 某一两个工具结果突然特别大。
2. 工具结果虽然不大，但累计很多。
3. 老结果已经不再重要，却还占着上下文。

这种时候直接走摘要压缩有点重，因为：

- 要额外调用模型。
- 会把原始上下文变成摘要，信息保真度下降。
- 不适合频繁在每一轮后都触发。

所以 defense 的定位更像“第一道处理层”：

- 先做便宜的、可预测的裁剪。
- 只动工具结果，不主动总结用户/assistant 主干对话。

## 三层防线分别做什么

### 第一层：Token Estimation

`estimateMessageTokens()` 会遍历 `ModelMessage[]`，把：

- 普通字符串消息
- text part
- tool result output

都转换成字符长度，再按大约 `4 chars = 1 token` 估算，并乘上一个中文安全系数。

这不是精确 tokenizer，但足够用来决定“要不要开始防御”。

### 第二层：Dynamic Tool Result Truncation

`truncateToolResults()` 分两步做。

第一步：单条结果截断

- 如果某个工具结果太长，就保留前 60% 和后 40%。
- 中间插入 `[truncated: ...]` 标记。

第二步：总预算清理

- 如果所有消息加起来仍然超过上下文预算，
- 就从最早的工具结果开始，把它们替换成
  `[compacted: <toolName> output removed to free context]`

也就是说，这一层既处理“单条太长”，也处理“总量太大”。

### 第三层：TTL Pruning

`ttlPrune()` 只处理 `tool` role，不处理 user/assistant。

它有两个时间阈值：

- `softTTLMs`：超过后做软修剪
- `hardTTLMs`：超过后做硬清除

规则是：

1. 如果工具结果像错误信息，就保留，不修剪。
2. 超过 soft TTL：
   - 保留头尾
   - 中间替换成 `[soft pruned: ...]`
3. 超过 hard TTL：
   - 直接替换成 `[tool result expired: <toolName>]`

这样旧的成功结果会逐渐淡出，但失败经验会尽量保留。

## `index.ts` 现在怎么接

当前接法在 `main()` 里完成，顺序是：

1. 初始化 `SessionStore`
2. 恢复 `messages`
3. 创建 `timestamps: Map<number, number>`
4. 用 `PromptBuilder` 组 system prompt
5. 每轮对话结束后：
   - 先把新增消息写入 session
   - 再执行 `runDefense()`
   - 如果 defense 改动了消息，就 `replaceAll()`

这里的关键点是 `timestamps`。

TTL 修剪不是看消息内容，而是看每条消息在 map 里记录的时间戳。所以在：

- 注入模拟历史时
- 用户新发消息时
- agent loop 新增 assistant/tool 消息时

都要同步更新 `timestamps`。

## 为什么要有 `sim / defend / status`

这套命令是为了把 defense 从“被动触发的内部逻辑”变成“可直接观测的测试入口”。

`sim` 会注入一段模拟长历史，特征是：

- 含多轮 user / assistant / tool 消息
- 含很大的 `read_file` 工具输出
- 时间分布有新有旧，方便测试 TTL

`defend` 会立即执行一轮 `applyDefense()`，打印：

- 截断了多少条
- 预算清理了多少条
- 软修剪了多少条
- 硬清除了多少条
- token 估算从多少降到多少

`status` 只负责查看当前状态，不改数据。

## 当前测试方法

建议直接这样测：

```bash
pnpm run start:mock
```

启动后输入：

```text
sim
status
defend
status
exit
```

预期现象：

1. `sim` 后消息数明显增加，token 估算显著变大。
2. `defend` 后会打印四类统计：
   - `截断`
   - `预算清理`
   - `软修剪`
   - `硬清除`
3. 第二次 `status` 里 token 会下降。

## 和摘要压缩的区别

当前 `defense.ts` 和 `compressor.ts` 不是一回事。

区别是：

1. `defense` 不依赖模型调用，成本低。
2. `defense` 只动工具结果，不主动总结整段历史。
3. `compressor` 适合长历史归纳，`defense` 适合每轮快速止血。

可以把它们理解成：

- `defense`：即时防线
- `compressor`：长期整理

## 当前已知限制

1. `estimateMessageTokens()` 仍然是估算，不是 provider 返回的真实 token。
2. TTL 依赖 `timestamps`，如果恢复历史时没有原始时间戳，只能用当前时间补齐。
3. 当前 defense 只覆盖工具结果，不会主动裁剪普通 assistant 文本。
4. `sim` 注入的是演示数据，不是模型真实跑出来的完整历史。

## 当前测试重点

这个阶段主要看三件事：

1. `index.ts` 能否在不接 MCP 的情况下独立跑起 defense 测试流。
2. `sim / defend / status` 三个入口能否稳定复现防线效果。
3. defense 执行后，session 文件能否通过 `replaceAll()` 保存当前防线结果。

建议验证命令：

```bash
pnpm run start:mock
pnpm run continue
```
