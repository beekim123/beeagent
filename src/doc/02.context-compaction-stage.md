# Context Compaction 阶段记录

这个阶段主要把会话持久化、system prompt 组装和上下文压缩接起来，目标是让对话在变长之后仍然可继续，而不是一直把完整历史塞给模型。

## 本阶段做了什么

1. 新增 `SessionStore`：
   - `append()`：把单条消息追加到 `.sessions/<sessionId>.jsonl`。
   - `appendAll()`：批量追加消息。
   - `load()`：从 jsonl 恢复历史消息。
   - `replaceAll()`：在压缩后整段重写 session 文件，保证磁盘里的历史和内存里的压缩结果一致。

2. 新增 `PromptBuilder`：
   - 把 system prompt 拆成多个 pipe。
   - 当前启用的 pipe 有：
     - `coreRules`
     - `toolGuide`
     - `deferredTools`
     - `sessionContext`
   - `PromptContext` 统一传入工具数量、延迟工具摘要、会话消息数和 sessionId。

3. 新增 `src/context/compressor.ts`：
   - `estimateTokens()`：粗略估算上下文 token。
   - `microcompact()`：先清理旧工具结果，降低体积。
   - `summarize()`：再用模型把更早的对话压成结构化摘要。
   - `CompactionResult`：返回压缩后的消息、摘要文本和被压缩的消息数。

4. `index.ts` 改成在 `main()` 里组装完整流程：
   - 先恢复 session。
   - 再从恢复内容里提取已有摘要。
   - 每轮对话前动态构建 system prompt。
   - 每轮结束后先写入 session，再检查是否需要压缩。
   - 压缩后把结果回写 session 文件。

## 压缩是怎么工作的

压缩分两层。

### 第一层：Microcompact

`microcompact()` 只处理旧的工具结果，不动对话主干。

它会扫描消息里的 `tool` role，优先保留最近 3 条工具结果，其他旧结果如果属于这些工具就替换成占位文本：

- `read_file`
- `bash`
- `grep`
- `glob`
- `list_directory`
- `edit_file`
- `write_file`

这一层的目的很直接：工具输出通常最长，先把最占 token 的部分降下来。

### 第二层：LLM Summarization

`summarize()` 会在 token 估算超过阈值后触发。

它的规则是：

1. 只压缩较早的对话。
2. 保留最近 6 条消息。
3. 先对齐到 `user` 消息边界，避免把一轮对话切断。
4. 把旧对话发给模型生成结构化摘要。
5. 用一条特殊的 `user` 消息把摘要塞回历史里，后续继续对话时还能读取。

摘要提示词要求模型输出这些字段：

- 用户意图
- 已完成的操作
- 关键发现
- 当前状态
- 需要保留的细节

## system prompt 现在怎么拼

`PromptBuilder` 不是把一整段固定 prompt 硬拼出来，而是按上下文动态拼接。

当前逻辑是：

1. `coreRules` 提供基础行为约束。
2. `toolGuide` 说明当前有多少工具可用。
3. `deferredTools` 提示去 `tool_search` 找延迟工具。
4. `sessionContext` 在会话已有历史时说明当前 sessionId 和消息数。

这样做的好处是：

- 会话为空时不会多塞无意义提示。
- 工具数量、延迟工具摘要、历史长度都会跟着当前状态变化。
- system prompt 的结构比手写字符串更容易扩展。

## 会话和压缩的关系

session 文件还是 jsonl，但现在不只是“对话日志”，也承载压缩后的历史。

当前流程是：

1. 用户消息先 `append()`。
2. `agentLoop` 追加 assistant/tool 消息后再 `appendAll()`。
3. 如果 token 超阈值，触发压缩。
4. 压缩后的 `messages` 通过 `replaceAll()` 覆盖写回。
5. 下次 `--continue` 时恢复到压缩后的状态。

这一步很关键，不然压缩只发生在内存里，下次恢复时还是原始长历史。

## 当前阈值

当前代码里有两个主要阈值：

- `COMPACTION_TRIGGER_TOKENS = 4000`
- `CONTEXT_TOKEN_THRESHOLD = 300`

前者控制什么时候开始检查压缩，后者控制 `summarize()` 自己是否真的执行压缩。

## 已知限制

1. `estimateTokens()` 只是粗略估算，不是模型真实 tokenizer。
2. `microcompact()` 只处理部分工具结果，属于保守裁剪。
3. `summarize()` 依赖当前模型生成摘要，摘要质量会受模型能力影响。
4. `replaceAll()` 会重写整个 session 文件，适合 CLI 这种单进程场景。

## 当前测试重点

这个阶段主要看四件事：

1. 新会话能否正常启动。
2. `--continue` 能否恢复压缩后的 session。
3. 超过阈值后能否打印压缩日志。
4. 压缩后 session 文件是否还能继续对话。

建议验证命令：

```bash
pnpm run start:mock
pnpm run continue
pnpm run typecheck
```
