# Tool Registry 阶段记录

这个阶段主要把工具从散落的对象改成统一注册和统一包装。最难理解的部分是 `ToolRegistry` 里加的异步读写锁，它的作用是：当模型并发触发多个工具时，允许安全工具一起执行，同时保护不安全工具独占执行。

## 本阶段做了什么

1. 新增 `ToolDefinition`，把每个工具需要的元数据集中描述：
   - `name`：工具名，也是暴露给模型调用的 key。
   - `description`：工具说明。
   - `parameters`：JSON Schema 参数定义。
   - `isConcurrencySafe`：工具是否可以和其他工具并发执行。
   - `isReadOnly`：工具是否只读，当前主要用于表达语义。
   - `maxResultChars`：工具结果最大返回长度。
   - `execute`：真正执行工具的函数。

2. 新增 `ToolRegistry`：
   - 负责注册工具。
   - 负责把 `ToolDefinition` 转成 AI SDK 需要的 tools 格式。
   - 统一包装工具执行逻辑，包括加锁、结果序列化、结果截断。

3. 新增集中工具列表 `allTools`：
   - `get_weather`
   - `calculator`
   - `read_file`
   - `write_file`
   - `list_directory`

4. `index.ts` 改成创建 registry 并注册 `allTools`，再把 registry 传给 `agentLoop`。

5. `agentLoop` 当前仍保持测试阶段逻辑：它只负责从 registry 转出 AI SDK tools 并交给 `streamText`，暂时不在 loop 层做更多调度优化。

## 为什么需要锁

AI SDK 支持 parallel tool calls。也就是说，模型可能在同一步里要求同时执行多个工具。

有些工具天然可以并发：

- `get_weather`：只查内存里的假天气数据。
- `calculator`：只计算表达式。
- `read_file`：只读文件。
- `list_directory`：只读目录。

但有些工具不应该和其他工具随便并发：

- `write_file`：会修改文件。如果它和另一个读写工具同时执行，可能出现读到旧内容、覆盖新内容、写入顺序不确定等问题。

所以 registry 里用 `isConcurrencySafe` 决定工具执行前拿哪种锁：

- `true`：拿共享锁，可以和其他共享工具一起跑。
- 其他值：拿独占锁，必须等所有工具结束后自己单独跑。

当前代码里：

```ts
const isSafe = tool.isConcurrencySafe === true;
```

这意味着只有明确标记 `isConcurrencySafe: true` 的工具才会走共享锁。未声明或声明为 `false` 的工具都会走独占锁，这是更保守的默认行为。

## 三个状态变量

`ToolRegistry` 里用三个变量实现读写锁：

```ts
private exclusiveLock = false;
private concurrentCount = 0;
private waitQueue: Array<() => void> = [];
```

它们分别表示：

- `exclusiveLock`：当前是否有独占工具正在执行。
- `concurrentCount`：当前有多少个共享工具正在执行。
- `waitQueue`：当前被阻塞的等待者。这里存的是 Promise 的 `resolve` 函数。

这不是操作系统线程锁，而是基于 Promise 的 JavaScript 异步锁。它控制的是同一个 Node.js 进程内多个 async tool 的进入顺序。

## 共享锁流程

共享锁用于并发安全工具。

```ts
private async acquireConcurrent(): Promise<void> {
  while (this.exclusiveLock) {
    await new Promise<void>(r => this.waitQueue.push(r));
  }
  this.concurrentCount++;
}
```

逻辑是：

1. 如果当前没有独占锁，直接进入。
2. 如果有独占锁，创建一个 Promise，把它的 `resolve` 放进 `waitQueue`，然后等待。
3. 被唤醒后重新检查 `while` 条件。
4. 真正能进入时，`concurrentCount++`。

多个共享工具可以同时进入，因为它们只要求没有独占锁。

释放共享锁时：

```ts
private releaseConcurrent(): void {
  this.concurrentCount--;
  if (this.concurrentCount === 0) this.drainQueue();
}
```

只有最后一个共享工具结束时，才唤醒等待队列。因为只要还有共享工具在执行，独占工具就不能进入。

## 独占锁流程

独占锁用于非并发安全工具。

```ts
private async acquireExclusive(): Promise<void> {
  while (this.exclusiveLock || this.concurrentCount > 0) {
    await new Promise<void>(r => this.waitQueue.push(r));
  }
  this.exclusiveLock = true;
}
```

逻辑是：

1. 如果已经有独占工具在执行，等待。
2. 如果还有共享工具在执行，等待。
3. 被唤醒后重新检查条件。
4. 真正能进入时，设置 `exclusiveLock = true`。

释放独占锁时：

```ts
private releaseExclusive(): void {
  this.exclusiveLock = false;
  this.drainQueue();
}
```

独占工具结束后，立即唤醒等待队列，让等待中的工具重新竞争锁。

## 为什么 drainQueue 是全部唤醒

```ts
private drainQueue(): void {
  const waiting = this.waitQueue.splice(0);
  for (const resolve of waiting) resolve();
}
```

这里不是唤醒一个，而是把等待者全部唤醒。

原因是唤醒不等于直接进入。每个等待者醒来后都会回到自己的 `while` 条件重新判断：

- 如果条件允许，它会拿锁进入。
- 如果条件不允许，它会再次把自己放回 `waitQueue`。

所以全部唤醒是一个简单可用的实现。代价是等待者多时会有一些重复唤醒，但当前阶段工具数量很少，够用。

## 一次执行的完整时间线

工具真正执行前，会先进入 registry 包装后的 `execute`：

```ts
execute: async (input: any) => {
  if (isSafe) {
    await registry.acquireConcurrent();
  } else {
    await registry.acquireExclusive();
  }

  try {
    const raw = await executeFn(input);
    const text = typeof raw === 'string' ? raw : JSON.stringify(raw, null, 2);
    return truncateResult(text, maxChars);
  } finally {
    if (isSafe) {
      registry.releaseConcurrent();
    } else {
      registry.releaseExclusive();
    }
  }
}
```

关键点：

1. 先根据 `isConcurrencySafe` 拿共享锁或独占锁。
2. 再执行原始工具函数 `executeFn(input)`。
3. 把非字符串结果转成 JSON 字符串。
4. 用 `truncateResult` 截断过长结果。
5. 在 `finally` 里释放锁。

`finally` 很重要。即使工具执行中抛异常，锁也一定会释放，否则后续工具可能永远卡住。

## 示例

假设模型同时请求：

1. `read_file`
2. `get_weather`
3. `write_file`

执行过程大概是：

1. `read_file` 标记为并发安全，拿共享锁，`concurrentCount = 1`。
2. `get_weather` 也标记为并发安全，也能拿共享锁，`concurrentCount = 2`。
3. `write_file` 不并发安全，想拿独占锁，但看到 `concurrentCount > 0`，进入等待队列。
4. `read_file` 结束，`concurrentCount = 1`，不唤醒队列。
5. `get_weather` 结束，`concurrentCount = 0`，唤醒等待队列。
6. `write_file` 重新检查条件，通过，设置 `exclusiveLock = true`，独占执行。
7. `write_file` 结束，释放独占锁，唤醒后续等待者。

## 结果截断

`truncateResult` 用来限制工具返回内容长度，避免大文件或长目录结果一次性塞给模型。

策略是：

- 没超过 `maxChars`：原样返回。
- 超过 `maxChars`：保留前 60% 和后 40%，中间加省略提示。

这样模型还能看到结果开头和结尾，比只保留前半段更有用。

## 当前阶段的已知限制

1. 这个锁只保护同一个 Node.js 进程内的工具执行，不是跨进程文件锁。
2. `waitQueue` 当前是全部唤醒再重新竞争，简单但不是最高效。
3. 如果未来一直有新的共享工具插进来，独占工具理论上可能等待较久。后续可以加 writer 优先级。
4. `isReadOnly` 目前主要是元数据，还没有参与调度策略。
5. 文件工具目前没有做路径沙箱限制，后续如果要真实使用写文件工具，需要补路径安全控制。

## 当前测试重点

这个阶段先测试三件事：

1. `index.ts` 能否把 `allTools` 注册进 `ToolRegistry`。
2. `agentLoop` 能否通过 `registry.toAISDKFormat()` 正常交给 AI SDK。
3. 工具执行时能否按 `isConcurrencySafe` 打印并发或串行日志，并正确返回结果。

基础验证命令：

```bash
pnpm typecheck
pnpm run start:mock
```

mock 模型当前主要覆盖天气、计算、重试和死循环测试。`read_file`、`write_file`、`list_directory` 这类文件工具更适合后续用真实模型或补 mock intent 后再测。
