# Usage Cost Tracking 阶段记录

这个阶段把每轮模型调用的 usage 统计、成本计算和 cache 视图接到了 `index.ts`，目标是让你能直接看见每一步花了多少 token、多少钱，以及 cache 命中情况。

## 本阶段做了什么

1. 新增 `UsageTracker`：
   - `record()`：记录单步 usage，并按模型价格表计算成本。
   - `totals()`：汇总 input、output、cache read、cache write、总成本和节省金额。
   - `recent()`：查看最近几步记录。

2. `agentLoop` 接入 usage 记录：
   - 每个 step 结束后，把 AI SDK 返回的 usage 规范化。
   - 再写入 `UsageTracker`。

3. `index.ts` 接入成本追踪视图：
   - 启动时创建 `.usage/today.jsonl`。
   - 增加 `/usage` 查看累计成本。
   - 增加 `/context` 查看当前上下文切片。
   - 增加 `/cache off` 和 `/cache on` 切换 mock 的 prompt cache。
   - 启动时会打印一次 context 视图，方便直接看当前状态。

4. 新增 `src/context/views.ts`：
   - `buildContextSnapshot()`：把 system prompt、工具描述、消息历史等估算成上下文切片。
   - `renderContextView()`：输出上下文矩阵。
   - `renderUsageView()`：输出成本、cache 命中率和节省金额。

## 成本是怎么计算的

`UsageTracker` 用 `src/usage/tracker.ts` 里的价格表计算：

- input tokens
- output tokens
- cache read tokens
- cache write tokens

同一模型下，cache hit 会按更低单价计算，所以 `/usage` 里会同时看到：

- `Cost`
- `Without cache`
- `Saved`

## 这次为什么要改 `index.ts`

因为这部分是 CLI 的入口，最适合把这些能力串起来：

- 创建 tracker
- 传给 `agentLoop`
- 暴露快捷命令
- 把上下文和 usage 打印出来

这样不用改业务逻辑文件，就能直接测试成本追踪。

## 怎么测试

### 1. 测 usage 汇总

启动：

```bash
pnpm run start:mock
```

然后输入几轮对话，再输入：

```text
usage
```

你应该能看到累计的 input/output token、cache 命中率和总成本。

### 2. 测 cache 开关

在 mock 模式下输入：

```text
cache off
```

再重复几轮相似输入，然后执行：

```text
usage
```

对比 `cache on` / `cache off` 的 `Cache read` 和 `Saved` 数值。

### 3. 测持久化

跑完后检查 `.usage/today.jsonl`，里面会有每一步的 usage 记录。

### 4. 测真实模型

如果配置了 `DASHSCOPE_API_KEY`，直接运行：

```bash
pnpm run start
```

再用同样的 `/usage` 看真实调用的累计成本。

## 已知限制

1. `windowTokens` 现在是估算用的固定值，不是 provider 真正返回的上下文上限。
2. mock 的 cache 开关只影响本地 mock 模型。
3. 费用表是按当前价格手工维护的，新增模型要同步更新 `src/usage/tracker.ts`。
